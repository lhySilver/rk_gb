# 需求背景: Issue Workflow 稳定性加固

## 背景
- 当前 `rk_gb` 已具备 GitHub issue 自动巡检和自动修复的工作流骨架。
- 上一轮实现重点完成了 workflow、triage 规则、repair 执行器和 RK830 隔离构建验证接入。
- 继续检查后发现，repair 队列在“无候选 issue”和“修复失败”两条路径上仍有明显稳定性缺口。

## 问题陈述

### 问题1: 无候选 issue 时定时 repair 会失败
`issue-repair.yml` 的定时任务固定执行 `--pick-next`。当前 `repair_executor.py` 在没有可处理候选时会直接抛错，导致 workflow 失败，而这在空队列场景应视为正常空转。

### 问题2: 修复失败 issue 会留在候选队列中反复被捞起
当前失败路径只添加 `ha-failed`，但不会移除 `ha-candidate`，`select_issue()` 也不会跳过 `ha-failed`。结果是同一个失败 issue 会被定时任务持续重复处理，产生失败风暴和重复评论。

### 问题3: 失败后的人工恢复路径没有写清楚
当自动修复失败后，需要人工判断是否重新打回候选、是否清理失败标签、是否继续 dry-run。但当前 README 和知识库对这条恢复路径说明不足。

## 目标
- 让 repair 定时任务在空队列场景下正常结束，不制造无意义失败告警。
- 让自动修复失败的 issue 默认退出候选队列，避免无限重试。
- 补充失败恢复和队列语义文档，便于后续仓库管理员和 runner 维护者使用。

## 非目标
- 不在本轮接入真实 GitHub Actions 线上验证。
- 不在仓库内实现真正的 AI 修复器本体。
- 不改变已有 triage 白名单/黑名单策略。
