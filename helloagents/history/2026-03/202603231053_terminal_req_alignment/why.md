# 变更提案: 终端需求与测试对齐

## 需求背景
当前知识库已经沉淀了 `GB28181` 和 `GAT1400` 的协议实现与规范基线，但仓库内新增了两份更贴近产品交付和验收的输入：
- `终端需求白皮书.pdf`
- `测试用例.xlsx`

这两份文档将国标基线进一步收敛为“中国移动定制版终端”的产品要求与验收用例，尤其覆盖了注册重定向、语音广播/对讲、远程配置、设备信息查询、画面翻转、远程升级等终端侧能力。若知识库只保留国家标准和代码实现，不补产品定制约束与测试映射，后续开发、联调和审核仍然容易出现“代码实现正确，但不满足白皮书/测试验收口径”的偏差。

## 变更内容
1. 提取白皮书中的终端侧 GB28181 产品要求与扩展协议要求。
2. 将测试用例按需求域进行归类，形成“需求条目 -> 测试用例 -> 当前实现/知识库入口”的对齐结果。
3. 补齐知识库中的终端需求对齐文档，并同步更新 `gb28181`、`api`、`data`、`overview`、`project` 等索引文档。

## 影响范围
- **模块:** GB28181、TerminalRequirements、项目级 API/数据模型/测试流程文档
- **文件:** `helloagents/wiki/modules/terminal_requirements.md`、`helloagents/wiki/modules/gb28181.md`、`helloagents/wiki/api.md`、`helloagents/wiki/data.md`、`helloagents/wiki/overview.md`、`helloagents/project.md`
- **API:** 无代码 API 变更，补充协议/产品要求说明
- **数据:** 无运行时数据结构修改，补充“白皮书字段与当前模型”的对照关系

## 核心场景

### 需求: 终端产品需求基线落库
**模块:** TerminalRequirements
需要把白皮书中的产品要求与国标基线分层表达，避免开发和审核只看 `GB/T 28181-2016` 而忽略产品定制扩展。

#### 场景: 白皮书中的广播/对讲和注册重定向要求可追溯
条件描述:
- 存在《中国移动定制版终端需求白皮书（2023 v1.0）》
- 存在与白皮书对应的测试用例
- 预期结果: 知识库中能直接查到注册重定向、语音广播/对讲、基础参数查询/配置、设备信息查询应答等产品要求
- 预期结果: 文档明确区分“国家标准要求”和“产品定制扩展要求”

### 需求: 测试用例与当前实现对齐
**模块:** GB28181
需要把测试用例回落到协议模块文档和数据模型文档，指出哪些能力已覆盖、哪些仍存在结构缺口或联调确认项。

#### 场景: 联调前可快速识别高风险缺口
条件描述:
- 测试用例覆盖注册、点播、控制、广播、对讲、参数配置、设备信息查询、升级
- 当前工程已具备 GB28181 App 层和 SDK 移植层
- 预期结果: 文档能指出 `VoiceFlowMode`、扩展 `DeviceInfo`、扩展 `BasicParam`、多码流数量等关键差异点
- 预期结果: 后续代码审核和联调优先检查这些差异点

## 风险评估
- **风险:** 把白皮书要求直接混入国家标准基线，导致概念混淆
- **缓解:** 单独新增 `terminal_requirements` 模块文档，并在 `gb28181` 中只补必要映射与注意事项
- **风险:** 只整理需求和测试，不回看当前实现，文档会缺乏指导价值
- **缓解:** 结合 `ProtocolManager`、`ProtocolExternalConfig`、`GB28181Defs`、`GB28181BroadcastBridge` 做最小实现对照
