# 技术设计: rk830 交叉编译隔离修复

## 技术方案

### 核心技术
- CMake out-of-tree 构建
- 命令级 `PATH` 注入本地 Rockchip 工具链
- 最小范围构建脚本或源码修复

### 实现要点
- 新建独立构建目录，例如 `rk_gb/build-rk830` 与 `rk_gb/Middleware/build-rk830`，不复用现有 `cmake-build`
- 使用 `env PATH="/home/jerry/silver/RK/arm-rockchip830-linux-uclibcgnueabihf/bin:$PATH" ...` 执行 `cmake` 与 `cmake --build`
- 若构建脚本只支持就地编译，则在不改变全局环境的前提下补充隔离构建入口
- 仅针对首个真实阻塞点做最小修复，避免无关重构

## 架构决策 ADR
### ADR-20260312-01: 采用命令级工具链注入替代全局环境配置
**上下文:** 本机存在其他 SoC 的编译任务，项目又将交叉编译器名硬编码在 `CMakeLists.txt` 中。  
**决策:** 不修改系统环境、不安装全局工具链、不持久化 `PATH`，而是在独立构建命令上临时注入 RK 工具链路径。  
**理由:** 可以同时满足“当前项目可编译”和“其他编译任务不受影响”两个约束。  
**替代方案:** 修改 `~/.bashrc` 或创建系统级软链接 → 拒绝原因: 会污染其他项目的编译环境。  
**影响:** 构建命令需要显式携带环境，但可复制、可审计、可回滚。

## 安全与性能
- **安全:** 不修改系统级环境变量，不清理或覆盖其他项目构建目录
- **性能:** 保持并行编译；如有必要只清理本次新建的 build 目录

## 测试与部署
- **测试:** 先执行 CMake 配置，再执行交叉编译，必要时重复至通过
- **部署:** 不做系统部署，仅验证编译产物生成
