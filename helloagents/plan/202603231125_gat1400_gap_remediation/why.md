# 变更提案: GAT1400完成度补齐

## 需求背景
当前仓库已经建立 `GA/T 1400.4-2017`、`GA/T 1400.3-2017`、`GA/T 1399` 的知识库基线，也已有一套可运行的 `GAT1400ClientService`，能完成注册、保活、校时、订阅接收、结构化对象上报和元数据/`Data` 双阶段上传。

但从“可审核”和“可交付”两个维度看，当前完成度仍然偏中间态：
- 更接近“下级客户端协议骨架”，不是“完整业务闭环”。
- 上报类 `Post*` 接口在仓库内未检索到明确业务侧调用路径，现状更像“能力已写好，事件源未接线”。
- 失败后的重试、重注册已经有基础逻辑，但缺少标准强调的“失败缓存、重新注册后补传”闭环。
- 当前传输层只解析 `http://`，若平台要求 `HTTPS` 或基于完整 URL 的接入方式，现有模型不够。
- 当前仓库未见完整 `/VIAS/*` 分析系统服务面，也未见 1400 相关自动化测试。

## 变更内容
1. 优先把现有 `lower client` 能力补到“联调可交付”状态，而不是直接扩成全量 VIID/VIAS 服务器。
2. 建立 `P0/P1/P2` 整改路线，区分联调阻塞项、增强项和远期扩展项。
3. 为后续代码实施明确文件边界、任务依赖和验证方式，避免只凭标准条文大范围改动。

## 影响范围
- **模块:** `GAT1400ClientService`、`ProtocolManager`、配置模块、`LowerGAT1400SDK` 对外适配面
- **文件:** `rk_gb/App/Protocol/gat1400/GAT1400ClientService.*`、`rk_gb/App/Protocol/ProtocolManager.*`、`rk_gb/App/Protocol/config/*`、`rk_gb/third_party/platform_sdk_port/CommonFile/CommonLib/LowerGAT1400SDK.h`
- **API:** `/VIID/System/*`、`/VIID/Subscribes*`、结构化对象上传、`/<ID>/Data` 上传、订阅通知
- **数据:** `GatRegisterParam`、`GatUploadParam`、订阅表、上传重试队列、注册状态

## 核心场景

### 需求: 注册与断链恢复闭环
**模块:** `GAT1400ClientService`
平台异常、链路超时或设备重启后，终端需要恢复注册状态，并继续发送保活和待上传数据。

#### 场景: 保活失败后重新注册
保活连续失败，或注册初次失败后延迟重试。
- 终端进入未注册状态
- 终端按配置策略重试注册
- 注册恢复后继续处理之前失败的待上传任务

### 需求: 结构化对象可真实上报
**模块:** 业务事件源 + `GAT1400ClientService`
人脸、人、车、物、场景、图片、文件等对象不仅要“有 SDK 接口”，还要真正接到业务生产路径。

#### 场景: 业务侧触发人脸和图片上报
设备产生抓拍结果。
- 业务事件转换为 `GAT_1400_*` 结构
- 调用 `PostFaces/PostImages` 进入上报链路
- 平台返回 `ResponseStatus` 后更新本地结果

### 需求: 订阅接收与通知闭环
**模块:** 本地 HTTP 服务、观察者接口
平台下发 `/VIID/Subscribes` 订阅后，终端要能被业务层感知并据此回发通知。

#### 场景: 平台新增订阅并收到通知
平台调用 `/VIID/Subscribes` 新增订阅。
- 终端本地服务接收并持有订阅
- 观察者或 SDK 导出接口把订阅变化传给业务层
- 后续匹配事件时可触发 `PostSubscribeNotifications`

### 需求: 传输与接入模型补齐
**模块:** 配置模块、HTTP 发送层
标准明确支持 HTTP，建议支持 HTTPS；同时联调环境常用完整 URL、非根路径和不同超时策略。

#### 场景: 平台要求 HTTPS 或完整 URL 接入
平台给出完整接入地址而非单纯 IP/Port。
- 配置能表达协议、主机、端口、路径和超时
- 发送层按配置组装请求
- 若平台要求 HTTPS，可明确支持或显式标注当前不支持

## 风险评估
- **风险:** 当前仓库未检索到 `LowerGAT1400SDK` 的完整导出实现，可能存在仓外二进制依赖。
- **缓解:** 实施前先确认 1400 对外 API 最终由本仓实现还是由外部库提供。
- **风险:** 失败缓存/补传会引入本地状态持久化，若边界没定义清楚，容易和现有即时重试逻辑冲突。
- **缓解:** 先抽象统一的待发送队列，再把所有 `Post*` 接口逐步迁移到同一重试模型。
- **风险:** `/VIAS/*` 能力范围很大，若与 lower client 完整度混在一起推进，周期和回归面会失控。
- **缓解:** 明确把 `/VIAS/*` 放到远期阶段，仅在平台明确要求时再启动。
