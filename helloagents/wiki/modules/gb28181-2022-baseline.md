## 3. 核心基线

### 3.1 注册、注销、重定向
- 注册和注销使用 `REGISTER`，并走挑战应答鉴权；典型链路是 `REGISTER -> 401 -> REGISTER(with Authorization) -> 200 OK`。
- 注册成功后需要在过期前刷新注册。标准给出的默认注册过期时间是 `86400s`，并要求不短于 `3600s`。
- 如果注册失败，下一次重试间隔一般不应短于 `60s`。所以连续秒级重注册通常不是标准期望行为。
- 开了 NAT 的设备宜使用 TCP 注册，并在注册成功后保持该 TCP 连接持续可用，后续心跳、刷新注册、点播、控制都复用这条连接。
- 支持 302 重定向时，若重定向后的 SIP 服务器离线或拒绝注册，应重新回到重定向流程，而不是一直死盯旧地址。

### 3.2 实时视音频点播
- 实时点播使用 `INVITE` 建立会话，媒体走 `RTP/RTCP`。
- 2022 版明确保留两种实现方式：客户端主动发起、第三方呼叫控制。工程调试时先分清平台走的是哪一种。
- SDP 中 `s=Play` 表示实时点播；`y` 用于 SSRC；`f` 描述媒体参数。
- 设备侧至少要把 `INVITE -> 200 OK -> ACK` 看成完整会话建立，不能在 ACK 前就把媒体链路判定为稳定完成。
- 音频实时点播同样属于 `Play`，但 `m=` 要落在 `audio`。如果平台只拉语音，不能按视频流逻辑去套。

### 3.3 控制、查询、通知的 MESSAGE 规则
- 设备控制、查询、状态信息报送大多采用 `MESSAGE`，`Content-Type` 是 `Application/MANSCDP+xml`。
- 对于收到的请求型 `MESSAGE`，设备应立即返回纯 SIP `200 OK`，这个 `200 OK` 本身不携带业务体。
- 是否还要再发一条业务应答 `MESSAGE`，取决于该类命令是不是“有应答命令流程”。
- 因此调试时要始终分开看两层：
  - 传输层确认：请求方是否收到纯 SIP `200 OK`
  - 业务层结果：是否又收到后续独立的 XML `MESSAGE`

### 3.4 状态信息报送 / Keepalive
- 心跳本质是状态信息报送，使用 `MESSAGE`。
- 默认心跳间隔 `60s`，默认超时次数 `3`；发送方和接收方需要配置一致。
- 连续超过超时次数没有收到响应时，双方都可以把对端判为离线。
- 如果抓包里 `Keepalive MESSAGE` 已经发出，但没有任何 `200 OK`，优先检查路由、TCP 连接复用、事务匹配和平台侧超时。

### 3.5 文件检索、历史回放、文件下载
- 文件检索请求的 `CmdType` 是 `RecordInfo`，请求体至少包含 `DeviceID`、`StartTime`、`EndTime`。
- 文件检索应答里要返回 `SumNum`。当没有任何结果时，标准要求 `SumNum=0`，并且不要携带 `RecordList`。
- 历史回放的 SDP 用 `s=Playback`，`u` 标识回放通道和类型，`t` 表示回放时间段。
- 文件下载的 SDP 用 `s=Download`，同样依赖 `u` 和 `t`；可扩展 `a` 字段表示下载倍速，设备侧也可在返回里带文件大小等扩展信息。
- 回放控制使用会话内 `INFO`，消息体走 `MANSRTSP`，用于暂停、恢复、快放、慢放、拖放等。
- 回放或下载结束后，发送侧还需要再发会话内 `MESSAGE` 做结束通知，不能只依赖 RTP 停流让平台自己猜。

### 3.6 校时
- 2022 版允许 SIP 校时和 NTP 校时，标准时间按北京时间处理。
- SIP 校时是在注册过程中完成的，关键依据是注册成功时最后一个 `200 OK` 里的 `Date` 头域。
- `Date` 头域格式为 `yyyy-MM-dd'T'HH:mm:ss.SSS`。
- 如果设备本地时钟漂移明显，标准建议把注册过期时间设到比“累计漂移 1 秒”更短的周期，以便通过刷新注册周期性校时。

### 3.7 订阅与通知
- 事件和目录订阅使用 `SUBSCRIBE`，变化通知使用 `NOTIFY`。
- 收到 `NOTIFY` 后，观察者需要先回 `200 OK`；通知异常处理遵循 RFC 6665。
- 目录、事件这类通知链路和广播链路类似，都要先保证 SIP 事务确认成功，再讨论业务消息体本身是否被接受。

### 3.8 语音广播
- 2022 版广播流程和很多旧实现不同，重点在于它不是“平台发通知，设备直接在同一个 `200 OK` 里塞业务 XML”。
- 标准链路应拆成四段：
  - 平台发 `Notify MESSAGE`
  - 设备先回纯 SIP `200 OK`
  - 设备独立发 `Broadcast` 应答 `MESSAGE`
  - 后续由设备侧主动发起音频 `INVITE`
- 广播的媒体协商是音频会话，抓包时要重点核对 `m=audio`、协商编码、传输方向以及后续 RTP 是否真的由设备按协商结果建链。

### 3.9 软件升级
- 软件升级属于控制类 `MESSAGE` 流程，收到平台请求后，设备侧仍应先完成纯 SIP `200 OK` 事务确认。
- 如果设备接受本次升级任务，可再独立返回控制结果 XML，表示本次命令已被受理；这不等同于升级已经成功完成。
- 最终升级结果应和实际执行结果绑定。对于需要下载、校验、写盘并重启的设备，最终成功通知应在升级完成并恢复上线后补报，而不是在“仅完成下载落盘”时提前上报成功。
- 工程调试时要分开看三层：
  - SIP `200 OK` 是否返回
  - 控制类业务应答是否为 `OK`
  - 最终 `DeviceUpgradeResult` 是否在执行完成后再单独上报

## 4. 对 rk_gb 的直接约束

### 4.1 代码落点
- `third_party/platform_sdk_port/CommonLibSrc/GB28181SDK/GB28181Client/GBClientImpl.cpp`
  - 负责 REGISTER、MESSAGE、INVITE 等 SIP 事务和 XML 打包/解析。
- `App/Protocol/ProtocolManager.cpp`
  - 负责把 GB 信令映射到本地媒体、录像、广播、配置和时间同步逻辑。
- `App/Protocol/gb28181/GB28181ClientReceiverAdapter.cpp`
  - 负责 SDK 回调与业务层之间的协议事件转接。
- `App/Protocol/gb28181/GB28181RtpPsSender.cpp`
  - 负责实时流/回放流的 PS/RTP 输出。
- `App/Protocol/gb28181/GB28181BroadcastBridge.cpp`
  - 负责广播音频接收、解包、播放和传输协商。
- `App/Protocol/config/LocalConfigProvider.*`
  - 负责 `gb28181.ini` 落盘与默认配置。

### 4.2 已知容易出错的点
- 把“纯 SIP `200 OK`”和“后续业务应答 `MESSAGE`”混成一次响应。
- 注册成功后没有按 `Date` 头域做时间对齐，或只打印日志不落地。
- `Playback/Download/Play` 的 `s=` 字段和真实业务不一致。
- 文件检索空结果仍携带空 `RecordList`，或 `SumNum`/列表条数不匹配。
- RTP over TCP 场景没有按协商结果处理 `setup:active/passive`，或者没有加帧长封装。
- 广播沿用旧版“等平台 INVITE”的思路，导致 2022 版平台不会继续后续流程。

## 5. 抓包排查 Checklist

### 5.1 注册问题
- 是否出现完整的 `401` 挑战应答。
- 刷新注册是否在过期前发出。
- 注册失败后的下一次重试是否过于频繁。
- NAT/TCP 场景下是否复用了同一条 TCP 通道。
- 注册成功 `200 OK` 是否带了 `Date`。

### 5.2 心跳问题
- `Keepalive MESSAGE` 是否定时发送。
- 平台是否返回 `200 OK`。
- 心跳序列号、设备编号是否与当前注册会话一致。
- 连续失败后是否按超时次数进入离线/重注册逻辑。

### 5.3 实时流/回放/下载问题
- `INVITE/200 OK/ACK` 是否成对完成。
- SDP 中 `s=` 是否和业务匹配：`Play`、`Playback`、`Download`。
- `m=`、端口、SSRC、`setup`、编码是否和平台协商一致。
- 媒体是在哪个时刻开始发的；如果 ACK 未到就发流，优先怀疑时序错误。
- 回放结束时是否有额外的结束通知 `MESSAGE`。

### 5.4 RecordInfo 问题
- 请求里的 `StartTime/EndTime/DeviceID` 是否正确。
- 应答的 `SumNum` 是否和条目数一致。
- 空结果时是否只返回 `SumNum=0`。
- 大量结果时是否需要拆批或多响应传输。

### 5.5 广播问题
- `Notify` 收到后是否先回纯 SIP `200 OK`。
- 是否单独发出了 `Broadcast` 应答 `MESSAGE`。
- 是否由设备主动发起了后续音频 `INVITE`。
- 广播会话中的 SDP 是否确实是音频协商。
- 建链成功后广播桥是否按协商端口、编码、传输方式收流。
